# 大QMT文件IPC Fallback 部署操作手册

> xttrader 降级替代方案的完整部署指南。当券商收紧 miniQMT 权限或 xttrader 连接失效时，
> 通过文件系统 IPC 把订单交给大QMT（完整客户端，自带 xttrader 授权）执行。

---

## 一、方案原理（30 秒理解）

```
┌──────── miniQMT 策略程序 ────────┐          ┌──────── 大QMT客户端 ────────┐
│  PositionManager / 策略 / 网格   │  文件IPC  │  qmt_trade_executor.py       │
│         ↓                        │  ═══════► │  (init 前台循环轮询)         │
│  QmtIpcTrader                    │          │         ↓                    │
│  buy/sell → 写 pending/*.json ───┼─────────►│  读指令 → xttrader 下单       │
│  position() ← 读 account.json ◄──┼──────────┤  写成交回执 → done/*.json    │
│                                  │          │  写账户快照 → account.json   │
└──────────────────────────────────┘          └──────────────────────────────┘
                     共享目录: C:\QuantIPC\
```

**关键点**：大QMT本身有 xttrader 授权，不依赖 miniQMT 权限。两个程序通过 JSON 文件通信，延迟约 1-2 秒，适合日频/小时频/分钟频策略。

## 二、前置条件

| 项 | 要求 |
|----|------|
| 大QMT客户端 | 已安装并能正常登录交易（完整版QMT，非miniQMT） |
| 大QMT内置Python | QMT自带Python策略编辑器，支持模型交易/Python策略脚本 |
| 磁盘 | 两端能访问同一个 `C:\QuantIPC\` 目录 |
| 账号 | miniQMT 的 `account_config.json` 已配好交易账号 |

> **跨机器部署**：若策略程序与大QMT在不同机器，把 `C:\QuantIPC\` 设为 SMB 网络共享，两端 `QMT_IPC_ROOT` 指向同一 UNC 路径（如 `\\SERVER\QuantIPC`）。

## 三、部署步骤

### 步骤 1：启用策略端开关


### 步骤 2：填写大QMT路径（一次性）

策略端已自动把账号写入 `C:\QuantIPC\{账号}\config.json`，但**大QMT安装路径需要你确认一次**。打开如 `C:\QuantIPC\TEST_ACC_1\config.json`：

```json
{
  "account_id": "TEST_ACC_1",
  "account_type": "STOCK",
  "qmt_path": "C:/QMT/userdata_mini",
  "updated_by": "miniQMT",
  "updated_at": "2026-07-09 22:30:00"
}
```

### 步骤 3：部署大QMT端执行脚本

1. 把 [qmt_trade_executor.py](qmt_trade_executor.py) 复制到大QMT策略编辑器目录（如 `user_define/`）
2. **无需手改脚本** — 脚本会自动扫描 `C:\QuantIPC\` 下所有账号子目录，读取各自的 `config.json`
   - 若 IPC 目录不是默认的 `C:\QuantIPC`，改脚本顶部 `IPC_ROOT` 常量
   - 多账号拓扑见第五章
3. 在大QMT中设置：**QMT → 策略交易 → Python策略 → 新建「模型交易」→ 选脚本**
   - 模型交易模式下 QMT 会自动注入 `get_trade_detail_data` 等必要 API
   - 光大金阳光 QMT 若找不到“定时运行”入口，直接使用截图里的“模型交易/实盘/运行”即可；新版 executor 会在 `init(ContextInfo)` 中进入前台循环，自行按 `QMT_IPC_MAIN_INTERVAL_SEC` 轮询 IPC
   - 不要勾选“启动本地 python”，否则可能跑到外部解释器，拿不到大QMT注入的交易上下文
4. 观察 `C:\QuantIPC\qmt_log\log_YYYYMMDD.txt`，应看到 `foreground loop started by init`、`tick ok accounts=... worker=False` 持续递增，以及各账号的 `snapshot xttrader: total=... positions=...`
5. 调试结论和避坑记录见 [大QMT能力探索与调试经验.md](大QMT能力探索与调试经验.md)

### 步骤 4：验证心跳

executor 运行后每秒更新 `C:\QuantIPC\{账号}\status\heartbeat.json`。策略端 `QmtIpcTrader.connect()` 检测到新鲜心跳即判定连接成功，日志中应看到 `QmtIpcTrader 连接成功`.

## 四、目录结构（多账号自动隔离）

```
C:\QuantIPC\
├── TEST_ACC_1\                    # 账号1 独立目录（进程1自动创建）
│   ├── config.json              # 该账号配置
│   ├── orders\ pending / processing / done
│   ├── cancel\
│   └── status\ heartbeat.json / account.json
├── TEST_ACC_2\                    # 账号2 独立目录（进程2自动创建）
│   └── ... (同上)
└── qmt_log\                     # 大QMT端日志（所有账号共享）
```

**隔离原理**：每个 miniQMT 账号是独立进程（通过 `QMT_ACCOUNT_ID` 环境变量选择），每个进程的 `QmtIpcTrader` 自动把工作目录隔离到 `{QMT_IPC_ROOT}/{account_id}/`，订单、回执、快照、心跳全部按账号隔离，互不串号。

## 五、多账号部署拓扑

### 拓扑A：同一大QMT登录多账号（推荐，一个 executor 搞定）

若多个资金账号在**同一个大QMT客户端**（同券商多账号），只需一个 executor：
- 自动扫描所有账号子目录
- 按 `qmt_path` 复用 xttrader 连接（同路径共享连接，多 `StockAccount` 下单）
- 无需额外配置

### 拓扑B：多个大QMT各管一账号

若不同账号在**不同大QMT**（如不同券商），每大QMT各跑一个 executor，用 `QMT_IPC_ACCOUNT_FILTER` 指定负责的账号：

```
# 大QMT-A（只处理 TEST_ACC_1）
set QMT_IPC_ACCOUNT_FILTER=TEST_ACC_1

# 大QMT-B（只处理 TEST_ACC_2）
set QMT_IPC_ACCOUNT_FILTER=TEST_ACC_2
```

> QMT 内置 Python 若不便设环境变量，可直接在脚本顶部把 `ACCOUNT_FILTER` 改成硬编码。

## 六、配置项参考（config.py）

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `ENABLE_QMT_IPC_FALLBACK` | `False` | 总开关，`.env` 中设 `true` |
| `QMT_IPC_ROOT` | `C:\QuantIPC` | IPC 目录，两端必须一致 |
| `QMT_IPC_ORDER_TIMEOUT` | `30` | 下单等待回执最大秒数 |
| `QMT_IPC_HEARTBEAT_MAX_AGE` | `10` | 心跳超时判定离线秒数 |
| `QMT_IPC_DEAL_POLL_INTERVAL` | `1.0` | 成交回报轮询间隔 |
| `QMT_IPC_DONE_LOOKBACK_SECONDS` | `86400` | done/ 委托查询回溯窗口 |

## 七、常见问题排查

### Q1：策略端日志报「大QMT心跳文件缺失或过期」
- 检查大QMT端 executor 是否处于“运行中”状态，并确认日志出现 `foreground loop started by init`
- 检查对应账号目录 `status/heartbeat.json` 的修改时间是否在 10 秒内
- 检查两端 `IPC_ROOT` / `QMT_IPC_ROOT` 是否指向同一目录
- 如果每次点击运行只刷新一次心跳，说明大QMT里运行的仍是旧脚本或策略没有重新编译保存；请更新到带前台循环的新版 `qmt_trade_executor.py`，重新编译、保存、运行

### Q2：大QMT端日志报「xttrader 连接失败」
- 检查 `config.json` 里 `qmt_path` 是否为大QMT的 `userdata_mini` 目录
- 检查 `account_id` 是否正确，大QMT客户端已登录交易

### Q3：下单一直超时返回 None
- 检查 `orders/pending/` 是否堆积未处理（大QMT端脚本未运行或崩溃）
- 查看 `qmt_log/` 里的错误
- 大QMT端每轮最多处理 5 个订单（防单轮超时），大量订单会排队

### Q4：持仓/资金显示为空
- `account.json` 每约 30 秒刷新一次，首次启动需等待
- 检查大QMT端 `write_snapshot()` 是否报错

### Q5：QMT脚本重启后 processing/ 有残留文件
- 无需手动操作: `recover_leftovers()` 自动将超龄文件(默认120秒)写 `error` 回执并清理, **绝不重发**以防止重复下单

### Q6：日志仍然高频刷 `tick start` / `tick done`
- 说明大QMT策略编辑器里运行的仍是旧版 executor。重新粘贴最新 `qmt_trade_executor.py` 并重启策略后，正常只会按 `QMT_IPC_MAIN_LOG_INTERVAL_SEC` 输出 `tick ok accounts=...` 摘要。

### Q6.1：日志只有一次 `tick ok ... ticks=1`，之后不再刷新
- 这是模型交易入口在非交易日或无新 K 线时只触发一次的典型表现。
- 新版 executor 会在 `init()` 中进入前台阻塞循环；正常日志应看到 `foreground loop started by init`，后续 `tick ok ... worker=False` 的 `ticks` 持续递增。
- 若没有这些日志，说明大QMT里运行的仍是旧脚本，或策略没有重新编译保存。

### Q7：休盘时资产为 0，但持仓有市值
- 属于 QMT/xttrader 常见休盘返回特征。executor 会过滤 `volume<=0` 的伪持仓，并在 `market_price=0`、`market_value>0` 时用 `market_value / volume` 反推参考价；`total_asset<=0` 时用 `available + market_value` 兜底。

### Q8：ContextInfo 里找不到交易 API
- 当前实测 `ContextInfo` 主要提供行情和上下文能力；`get_trade_detail_data`、`passorder`、`cancel` 注入在 `globals()`。下单读写仍应优先使用 `xttrader + StockAccount`，VBA 作为 fallback。

### Q9：订单 JSON 卡在 processing/，日志提示 BOM 或解析失败
- Windows PowerShell 的 `Set-Content -Encoding UTF8` 可能写出 UTF-8 BOM。新版 executor 已用 `utf-8-sig` 读取 IPC JSON，可直接兼容。
- 如果 JSON 本身损坏，executor 会立即写 `done/error` 并清理 `processing/`，不会再等 120 秒残留恢复。
- 若仍卡住，优先确认大QMT里运行的是最新版 `qmt_trade_executor.py`，并查看 `qmt_log/` 中对应订单号的错误。

### Q10：空仓卖出或柜台拒单后返回 rejected
- 这是正常保护结果，不代表 IPC 通道失败。
- 已验证空仓卖出时 `order_stock` 可能直接返回 `-1`，executor 会写 `done/rejected` 且 `filled_volume=0`。
- 若 QMT 先返回有效委托号、随后在委托查询中返回废单状态 `57`，executor 同样会写 `done/rejected`。

## 八、回退（恢复 xttrader 直连）

编辑 `.env`：
```dotenv
ENABLE_QMT_IPC_FALLBACK=false
```
重启 miniQMT。工厂函数恢复返回 `easy_qmt_trader`（xttrader 直连），行为与未部署时完全一致。

## 九、能力边界与限制

| 维度 | 说明 |
|------|------|
| **延迟** | 下单到成交回执约 1-2 秒；适合中低频，不适合高频/打板 |
| **持仓实时性** | 数量/成本来自快照（~30秒刷新）；实时价仍由 xtdata/Mootdx 提供 |
| **委托查询** | 用 done/ 目录回执近似构造，精度低于 xttrader 原生查询 |
| **单轮吞吐** | 每账号每轮最多 5 个订单，防止单次定时任务超时 |
| **适用策略** | 止盈止损、网格、自动买入等日频/分钟频策略；不适合 tick 级 |
